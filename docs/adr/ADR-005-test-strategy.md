# ADR-005: Test Strategy — Characterization Tests on AngularJS, Unit Tests on Migrated Angular Code

**Status:** Accepted  
**Date:** 2026-04-24  
**Deciders:** Frontend team, QA

---

## Context

The AngularJS codebase has no automated tests. Migration without tests means regressions cannot be detected. We need a test strategy that:

1. Pins the current AngularJS behavior before it is changed ("characterization tests").
2. Provides unit tests for each migrated Angular component/service/pipe.
3. Does not require the entire app to be migrated before tests run.

---

## Decision

We use two layers of testing, applied at different points in the migration:

### Layer 1 — Characterization Tests (pre-migration, on AngularJS code)

Written **before** a file is migrated. They capture what the code currently does — bugs included. Their purpose is not to verify correctness, but to detect unintentional behavior changes during migration.

```ts
// Example: pinning an AngularJS filter's current behavior
describe('legacyDateFilter (characterization)', () => {
  it('formats 20240101 as "Jan 1, 2024"', () => {
    // current behavior — even if the format seems odd
    expect(legacyDateFilter('20240101')).toBe('Jan 1, 2024');
  });
  it('returns empty string for null input (bug: should probably throw)', () => {
    expect(legacyDateFilter(null)).toBe('');  // pinning the bug
  });
});
```

If a characterization test fails after migration, it means behavior changed — investigate before proceeding.

### Layer 2 — Unit Tests (post-migration, on Angular code)

Generated by the `generate-tests` skill immediately after each file is migrated. These test the Angular component/service/pipe in isolation using `TestBed`.

Targets per type:
- **Services**: one test per public method; HTTP calls use `HttpTestingController`; signal state changes verified with `signal()` reads.
- **Components**: one test per `@Input`, one per `@Output`, one per user interaction.
- **Pipes**: happy path + edge cases (null, empty, unexpected types).

### Test Framework

Detect from `package.json`: Jest preferred, Jasmine/Karma if already configured.

---

## Migration Test Workflow

```
For each file to migrate:
  1. Write characterization tests for the AngularJS file
  2. Run characterization tests — confirm they pass
  3. Migrate the file using the appropriate skill
  4. Run generate-tests skill → produces Angular unit tests
  5. Run both suites — both must be green on the same commit
  6. (Optional) Delete characterization tests once the Angular unit tests cover the same behavior
```

---

## What We Chose Not to Do

| Option | Reason rejected |
|--------|----------------|
| No tests during migration ("add tests later") | Regressions accumulate silently; "later" never comes |
| E2E tests only (Cypress/Playwright) | Too slow for the migration feedback loop; cannot isolate which file broke |
| Rewrite tests from scratch without pinning first | We lose visibility into whether migration changed behavior |
| 100% coverage requirement | Not realistic during migration; focus on public API behavior, not line coverage |

---

## Consequences

**Positive:**
- Any migration that changes behavior is caught immediately, on the same commit.
- Characterization tests document implicit contracts that were never written down.
- The `generate-tests` skill produces consistent test scaffolding, reducing the cost of writing tests.

**Negative:**
- Writing characterization tests for AngularJS code requires understanding the AngularJS test harness (`$injector`, `$compile`, `$rootScope.$digest`).
- Some AngularJS behavior (e.g., deep `$watch` side effects) is hard to characterize without running the full app.
- The test suite grows in two phases; the team must maintain both until characterization tests are retired.

**Skill to use:** `.agent/skills/generate-tests/SKILL.md` after every file migration.
